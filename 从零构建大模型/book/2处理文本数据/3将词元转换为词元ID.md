1. **构建词汇表**(token → integer 的字典)
2. **构建逆向词汇表**(integer → token,用于解码)
3. **封装成 `SimpleTokenizerV1` 类**,提供 `encode` 和 `decode` 方法
4. **发现一个致命问题**——这个分词器遇到训练集没见过的词就崩——为 2.4 节的"特殊词元"埋伏笔

---

# 2.3 将词元转换为词元 ID

## 一、为什么需要这一步

**神经网络只吃数字**。上一节我们把文本切成了字符串列表:

```python
['I', 'HAD', 'always', 'thought', 'Jack', ...]
```

但 `'Jack'` 是字符串,神经网络不能直接处理。我们要把每个词元映射成**一个整数 ID**:

```python
['I', 'HAD', 'always', ...]  →  [40, 367, 245, ...]
```

⚠ 注意:整数 ID 还不是嵌入向量。这只是**中间步骤**——之后(2.7 节)我们会用 ID 去**查表**得到嵌入向量。所以流程是:

```
文本 → 词元(字符串) → 词元 ID(整数) → 嵌入向量(浮点数列表)
       ↑ 2.2 节          ↑ 2.3 节        ↑ 2.7 节
```

---

## 二、构建词汇表:三行代码

```python
all_words = sorted(set(preprocessed))           # 去重 + 排序
vocab_size = len(all_words)                     # 1130
vocab = {token: i for i, token in enumerate(all_words)}
```

**拆解每一步**:

| 操作                | 作用                                  |
| ------------------- | ------------------------------------- |
| `set(preprocessed)` | 去重——同一个词只保留一次              |
| `sorted(...)`       | 按字母顺序排列(确定性,跨机器结果一致) |
| `enumerate(...)`    | 给每个唯一词元配一个从 0 开始的整数   |

The Verdict 全文有 **4690 个词元**,但去重后只有 **1130 个唯一词元**——这就是词汇表的大小。

> 直觉:小说里 "the"、"a"、"," 这些会出现成百上千次,但作为词元只算一个。

---

## 三、为什么还要逆向词汇表?

LLM 的工作流程是**双向**的:

```
用户输入文本  →  编码(encode)  →  词元 ID  →  模型计算
                                                    ↓
用户看到回复  ←  解码(decode)  ←  词元 ID  ←  模型输出
```

模型**输出的也是词元 ID**,我们要把 ID 还原成人能读的文本——这就需要反向映射:

```python
self.int_to_str = {i: s for s, i in vocab.items()}  # 反转字典
```

---

## 四、`SimpleTokenizerV1` 类详解

```python
class SimpleTokenizerV1:
    def __init__(self, vocab):
        self.str_to_int = vocab                              # 正向表
        self.int_to_str = {i: s for s, i in vocab.items()}  # 反向表

    def encode(self, text):
        # 1. 用 2.2 节的正则切词
        preprocessed = re.split(r'([,.?_!"()\']|--|\s)', text)
        preprocessed = [item.strip() for item in preprocessed if item.strip()]
        # 2. 查表:字符串 → 整数
        ids = [self.str_to_int[s] for s in preprocessed]
        return ids

    def decode(self, ids):
        # 1. 查反向表:整数 → 字符串,用空格拼起来
        text = " ".join([self.int_to_str[i] for i in ids])
        # 2. 修补:删除标点前的空格,例如 "hello ," → "hello,"
        text = re.sub(r'\s+([,.?!"()\'])', r'\1', text)
        return text
```

### `decode` 里的小细节

注意 `re.sub(r'\s+([,.?!"()\'])', r'\1', text)`——这一步很容易被忽略,但**没有它输出会很难看**。

为什么?因为 `decode` 用空格 `" ".join(...)` 拼接,但标点前**不该有空格**:

```
不修补:  "Hello , world ."
修补后:  "Hello, world."
```

正则 `\s+([,.?!"()\'])` 意思是"标点前的空白",`\1` 表示"保留括号里捕获的标点本身,但去掉前面的空白"。

## 五、实际跑一遍

我把小说 The Verdict 拼好(用真实的内容前段),完整跑一遍 2.3 节:全部跑通 ✓。来对照书上的输出:

| 检查项                               | 书上                                | 我们的结果                          | 说明                                         |
| ------------------------------------ | ----------------------------------- | ----------------------------------- | -------------------------------------------- |
| 词汇表前几项                         | `('!', 0), ('"', 1), ("'", 2), ...` | `('"', 0), ("'", 1), ('(', 2), ...` | 略不同,因为我用的是片段不是全文,所以没有 `!` |
| 编码 `"It's the last..."` 的 ID 长度 | 21 个数字                           | 21 个数字 ✓                         | 序列结构一致                                 |
| `Hello` 触发 KeyError                | ✓                                   | ✓                                   | 完美复现                                     |

注意 decode 的输出有个小瑕疵: `" It' s the last..."` ——`It's` 被拆成了 `It` + `'` + `s`,reconstruct 时 `'` 和 `s` 之间多了个空格。

**这是分词器的固有局限**:正则把 `'` 当分隔符,所以 `It's` 在分词时永远会被切成 3 个词元,decode 时无法恢复成原样。后面的 BPE 分词器(2.5 节)会更聪明地处理这种情况。

## 六、致命问题:词汇表是封闭的

最后那个 `KeyError: 'Hello'` 不是 bug,是设计缺陷。

我们的词汇表是基于 **The Verdict 这一篇小说** 构建的——只有 1130 个词。但真实世界的文本里有无穷多个词:`Hello`、`tea`、`smartphone`、`COVID-19`、人名、新造词……

**两个思路解决**:

| 思路                                             | 做法           | 章节                |
| ------------------------------------------------ | -------------- | ------------------- |
| ① 把所有"陌生词"统一记为一个特殊词元 `<\|unk\|>` | **2.4 节**     | 简单粗暴,但损失信息 |
| ② 把陌生词拆成已知的子词或字符                   | **2.5 节 BPE** | GPT 实际采用的方案  |

---

## 八、自测

1. 词汇表为什么要 `sorted`?乱序行不行?
2. `decode` 方法里为什么要专门写一行正则去删标点前的空格?
3. 如果不构建 `int_to_str` 反向词汇表,会怎样?
4. 我们的词汇表大小是 1130。如果在大规模英文语料(比如整个维基百科)上构建,词汇表会有多大?这有什么问题?

<details>
<summary>参考答案</summary>

1. **可行,但不推荐**。乱序的话每次构建的 vocab 都不一样,同一个词在不同次运行中 ID 不同——无法复现、无法分享模型权重。`sorted` 保证了**确定性**。

2. 因为 `decode` 用 `" ".join(...)` 拼接所有词元,标点也被空格隔开了,变成 `"hello , world ."`。这看起来很丑,正则把标点前的空格删掉,恢复成 `"hello, world."`。

3. 模型输出的是整数 ID,如果没有反向表,就无法把数字还原成人能读的文本——用户看不懂模型说什么。

4. **几十万到上百万**。问题:① 嵌入矩阵会爆炸(vocab_size × embedding_dim,12288 维 × 100 万词 ≈ 122 亿参数,光嵌入层就比小 GPT-2 还大);② 长尾词出现次数少,学不到好的嵌入;③ 总会有新词冒出来(网络流行语、专有名词、新产品名)。BPE 通过"子词"思路同时解决这三个问题。

</details>
